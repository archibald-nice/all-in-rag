## 🔥 产品经理核心能力清单

### 1️⃣ 技术理解（不做开发，但要懂边界）

| 能力 | 表现 |
|-----|------|
| **解释 RAG** | 能用通俗语言向老板/客户解释 RAG 是什么 |
| **识别适用场景** | 判断什么业务适合用 RAG |
| **评估可行性** | 知道什么能做、什么做不了、成本如何 |
| **沟通协作** | 能和开发对齐需求，不被技术术语吓到 |

### 2️⃣ 产品设计（基于技术能力做决策）

| 决策点 | 考虑因素 |
|-------|---------|
| **检索策略** | 向量？关键词？混合？→ 根据用户查询特点 |
| **分块策略** | 块大小？重叠多少？→ 根据文档类型和查询粒度 |
| **生成风格** | 严谨专业？亲切友好？→ 根据品牌调性和用户群 |
| **功能优先级** | 基础检索 → 引用来源 → 多轮对话 → 个性化 |

### 3️⃣ 用户体验（把技术转化为体验）

| 体验要素 | 设计要点 |
|---------|---------|
| **提问引导** | 示例问题、自动补全、查询重写 |
| **结果呈现** | 引用来源、结构化输出、相关推荐 |
| **错误处理** | 没找到答案时怎么办？AI 瞎编时怎么提示？ |
| **反馈机制** | 点赞/点踩、反馈收集、持续优化 |

## 💼 实战练习（产品经理）

### 练习 1：产品设计文档
选择一个你熟悉的业务场景（比如：企业知识库、法律咨询、医疗问诊），回答：
1. 适合用 RAG 吗？为什么？
2. 核心用户场景是什么？
3. 需要什么样的数据？如何分块？
4. 如何衡量产品好坏？

### 练习 2：竞品分析
找一个 RAG 产品（如：Perplexity、客服机器人、知识库系统），分析：
1. 它的检索策略是什么？（能感觉出来吗？）
2. 它如何展示引用来源？
3. 用户体验有什么亮点/不足？

### 练习 3：技术评审
假设开发说："我们需要用混合检索 + 重排序 + 图 RAG"，你要：
1. 理解这些技术是干什么的
2. 问对问题：为什么需要？成本多少？有没有更简单的方案？
3. 做出产品决策：是否值得投入？

## ⚠️ 产品经理常见误区

| 误区 | 正确认知 |
|-----|---------|
| "RAG 能解决所有问题" | ❌ RAG 不是万能的，知识库质量决定上限 |
| "块越大越好" | ❌ 块太大会导致检索不准 |
| "向量检索最先进" | ❌ 很多场景关键词检索更好用 |
| "AI 越贵越好" | ❌ 要根据场景选择，不是所有场景都需要 GPT-4 |
| "一次开发就完成" | ❌ RAG 系统需要持续优化数据和提示词 |

---

## 🎯 总结：产品经理的 3 个核心问题

在评估任何 RAG 产品时，问自己：

1. **技术可行性**：这个业务适合用 RAG 吗？有什么技术限制？
2. **产品价值**：能解决什么用户痛点？和现有方案比有什么优势？
3. **投入产出比**：需要多少成本（数据、开发、模型）？收益有多大？

---

**记住**：作为产品经理，你的目标是理解技术边界，做出正确的产品决策，而不是自己写代码。🚀
